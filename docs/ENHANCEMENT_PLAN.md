## 2. AI/ML 기반 지능형 서비스

### 2.1 어르신 컨디션 예측 시스템

#### 개요
일과 보고, 복약 기록, 활동 참여, 식사량 등의 데이터를 분석하여 어르신의 컨디션 변화를 조기에 감지하고 예측

#### 기술 구현

```
[데이터 수집]                    [ML 파이프라인]                [알림/조치]

일과보고 (컨디션, 기분, 식욕) ──►┐
복약 기록 (준수율, 이상반응) ──►├─► 특성 추출 ─► 모델 학습 ──► 컨디션 예측
활동 참여 (참여율, 집중도)  ──►├─► (Python)    (XGBoost/     (정상/주의/경고)
식사량 기록              ──►├─►              LSTM)              │
바이탈 데이터 (IoT 연동)  ──►┘                              ┌──┘
                                                           │
                                                     주의 이상 시
                                                     ├── 시설 직원 알림
                                                     ├── 보호자 알림
                                                     └── 건강 리포트 자동 생성
```

#### 예측 모델

| 모델 | 입력 데이터 | 출력 | 용도 |
|------|------------|------|------|
| 컨디션 예측 | 최근 7일 일과보고 + 복약 + 활동 | 향후 3일 컨디션 | 조기 이상 감지 |
| 낙상 위험도 | 활동량, 보행 패턴, 복약 | 낙상 위험 점수 | 예방 조치 |
| 인지기능 변화 | 인지훈련 점수, 소통 빈도, 활동 참여 | 인지 변화 추이 | 치매 진행 모니터링 |
| 복약 이탈 예측 | 과거 복약 패턴, 부작용 기록 | 이탈 확률 | 사전 개입 |

#### 구현 기술 스택

| 항목 | 기술 | 비고 |
|------|------|------|
| ML 프레임워크 | Python (scikit-learn, XGBoost) | 초기 모델 |
| 딥러닝 | TensorFlow Lite | 시계열 예측 (LSTM) |
| 서빙 | Supabase Edge Functions + Python 모델 서버 | REST API |
| 스케줄링 | pg_cron (Supabase) | 일 1회 배치 예측 |
| 온디바이스 | TFLite Flutter 플러그인 | 간단한 이상 감지 |

### 2.2 자연어 처리 (NLP) 활용

| 기능 | 설명 | 기술 |
|------|------|------|
| 일과보고 자동 요약 | 직원이 메모 입력 → AI가 구조화된 보고서 자동 생성 | GPT API / Claude API |
| 감정 분석 | 채팅 메시지에서 보호자/직원 감정 분석 → 만족도 자동 측정 | 감성 분석 모델 |
| 음성 입력 | 직원이 음성으로 보고서 작성 → STT + 자동 구조화 | Google STT + LLM |
| 챗봇 | 보호자 FAQ 자동 응답 (시설 운영시간, 면회 안내 등) | RAG 기반 챗봇 |

### 2.3 컴퓨터 비전 활용

| 기능 | 설명 | 기술 |
|------|------|------|
| 복약 사진 자동 검증 | 촬영 사진에서 약 종류/개수 자동 확인 | 이미지 분류 모델 |
| 활동 사진 자동 태깅 | 프로그램 사진 자동 분류 (미술, 음악, 운동 등) | 이미지 분류 |
| 표정 분석 | 어르신 사진에서 기분/컨디션 보조 판단 | Facial Expression Recognition |

## 3. IoT 디바이스 연동

### 3.1 연동 가능 디바이스

```
┌─────────────────────────────────────────────────────────┐
│                    SilverSync IoT 생태계                  │

│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ 스마트밴드 │  │ 혈압계    │  │ 혈당계    │              │
│  │(심박,걸음)│  │(BLE 연동)│  │(BLE 연동)│              │
│  └─────┬────┘  └─────┬────┘  └─────┬────┘              │
│        │             │             │                    │
│        └─────────────┼─────────────┘                    │
│                      ▼                                  │
│              ┌──────────────┐                            │
│              │  BLE Gateway  │                            │
│              │ (Flutter 앱)  │                            │
│              └──────┬───────┘                            │
│                     ▼                                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
```

### 3.2 디바이스별 기능

| 디바이스 | 수집 데이터 | 활용 | 우선순위 |
|----------|------------|------|----------|
| **스마트밴드** | 심박수, 걸음수, 수면 | 건강 모니터링, 활동량 추적 | ★★★ |
| **전자 혈압계** (BLE) | 수축/이완 혈압, 맥박 | 자동 건강 기록, 이상치 알림 | ★★★ |
| **전자 혈당계** (BLE) | 혈당 수치 | 당뇨 관리, 식단 연계 | ★★☆ |
| **체온계** (비접촉) | 체온 | 발열 감지, 감염 조기 경고 | ★★☆ |
| **실내 비콘** (BLE) | 실내 위치 | 배회 감지, 활동 동선 분석 | ★★☆ |
| **침대 센서** | 움직임, 이탈 | 낙상 감지, 수면 질 분석 | ★☆☆ |
| **약 디스펜서** (스마트) | 투약 시간, 잔여량 | 자동 복약 기록, 재처방 알림 | ★☆☆ |

### 3.4 IoT 데이터 파이프라인

```
디바이스 → BLE → Flutter 앱 → Supabase (health_records)
                                    │
                                    ├── Realtime → 보호자 앱 (실시간 대시보드)
                                    ├── Edge Function → 이상치 감지 → 푸시 알림
```

### 3.5 추가 DB 스키마

```sql
CREATE TABLE health_records (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  resident_id UUID REFERENCES residents(id),
  device_type TEXT NOT NULL,          -- blood_pressure, glucose, heart_rate, temperature, steps
  value JSONB NOT NULL,               -- {"systolic": 130, "diastolic": 85, "pulse": 72}
  measured_at TIMESTAMPTZ NOT NULL,
  device_id TEXT,                     -- BLE 디바이스 ID
  recorded_by UUID,                   -- 자동(NULL) 또는 직원(staff ID)
  is_abnormal BOOLEAN DEFAULT false,  -- 이상치 여부
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 이상치 기준 설정 테이블
CREATE TABLE health_thresholds (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  resident_id UUID REFERENCES residents(id),
  metric TEXT NOT NULL,               -- systolic, diastolic, glucose, heart_rate, temperature
  min_value REAL,
  max_value REAL,
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(resident_id, metric)
);
```

## 4. 데이터 분석 플랫폼

### 4.1 시설 운영 분석 대시보드

| 분석 항목 | 지표 | 활용 |
|----------|------|------|
| **이용률 분석** | 일/주/월별 이용자 수, 이용률 추이 | 정원 관리, 인력 배치 최적화 |
| **프로그램 효과** | 프로그램별 참여율, 만족도, 인지기능 변화 | 프로그램 개선 |
| **복약 준수율** | 시설별/어르신별 준수율, 미복약 사유 분석 | 복약 관리 개선 |
| **보호자 만족도** | 소통 빈도, 응답 시간, 앱 사용 패턴 | 서비스 품질 개선 |
| **직원 업무량** | 기능별 사용 빈도, 보고서 작성 시간 | 업무 효율화 |

### 4.2 벤치마킹 리포트

```
┌─────────────────────────────────────────┐
│      시설 운영 벤치마킹 리포트 (월간)     │
```

## 6. 기술 인프라 고도화

### 6.1 성능 최적화

| 영역 | 현재 | 고도화 | 기대 효과 |
|------|------|--------|----------|
| 위치 데이터 | 5초 간격 INSERT | TimescaleDB 시계열 DB | 10배 쿼리 성능 |
| 이미지 | 원본 업로드 | CDN + 자동 리사이징 | 로딩 70% 개선 |
| 실시간 | Supabase Realtime | WebSocket 클러스터 | 동시접속 10배 |
| 캐싱 | 없음 | Redis 캐시 레이어 | API 응답 80% 개선 |
| 검색 | PostgreSQL LIKE | Elasticsearch / pg_trgm | 풀텍스트 검색 |

### 6.2 마이크로서비스 전환 (장기)

```
현재 (모놀리식)                  고도화 (마이크로서비스)

┌──────────────┐               ┌──────────┐  ┌──────────┐
│   Supabase   │               │ Auth     │  │ 송영     │
│  (All-in-One)│       ───►    │ Service  │  │ Service  │
└──────────────┘               └──────────┘  └──────────┘
                               ┌──────────┐  ┌──────────┐
                               │ 복약     │  │ 소통     │
                               │ Service  │  │ Service  │
                               └──────────┘  └──────────┘
                               ┌──────────┐  ┌──────────┐
                               │ 분석     │  │ IoT      │
                               │ Service  │  │ Service  │
                               └──────────┘  └──────────┘
```

> 주의: 마이크로서비스 전환은 시설 1,000곳 이상, 월 처리량이 Supabase 한계에 도달할 때 검토. 조기 전환은 오버엔지니어링.

## 9. 글로벌 확장 전략

### 9.1 1차 목표: 일본 시장

| 항목 | 내용 |
|------|------|
| 시장 규모 | 개호(介護) 시장 약 12조엔 (2025) |
| 시설 수 | 특별양호노인홈 약 10,000개, 주간서비스 약 44,000개 |
| 진입 전략 | 현지 개호 소프트웨어 기업과 파트너십 |
| 현지화 | 일본어 UI + 개호보험 제도 연동 + 일본식 UI 패턴 |
| 규제 | 개인정보보호법(APPI), 개호보험법 준수 |

### 9.2 2차 목표: EU 시장

| 항목 | 내용 |
|------|------|
| 시장 | 독일, 네덜란드, 스웨덴 우선 |
| 규제 | GDPR 준수 필수 |
| 현지화 | 다국어 (영어, 독일어, 네덜란드어) |
